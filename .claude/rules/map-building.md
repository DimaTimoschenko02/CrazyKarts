# Map Building Rules

Правила и факты про сборку карт из Kenney Space Kit (`assets/map_materials/space-kit/*.glb`). Накапливаем по мере экспериментов — каждый факт подтверждён скриншотом/тестом.

## Основная грид-система

- **1 cell = 1×1 метр в плане** (tile_size=1.0)
- **level_height = 0.5 метра** (один уровень = 0.5м вверх)
- **BBox ассета = логическая cell**, но видимая геометрия может занимать только часть cell (остаток — прозрачный воздух).

## Пивоты ассетов: починены import script'ом

Все Kenney Space Kit `.glb` экспортированы из одной большой Blender-сцены, поэтому их вершины сидят далеко от локального origin (типичный offset — `(2.0, _, 1.5)` от визуального центра). Проблема решена раз навсегда через **EditorScenePostImport**:

- `tools/recenter_on_import.gd` — import hook, который при каждом импорте `.glb` считает AABB и смещает корневого child'а так, чтобы визуальный центр XZ лёг в `(0, y, 0)`
- Все 153 `.glb.import` файла в `assets/map_materials/space-kit/` ссылаются на этот скрипт через `import_script/path="res://tools/recenter_on_import.gd"`
- После импорта: AABB каждого ассета центрирован по XZ. Пивот = визуальный центр нижней грани cell
- Y **не трогается** — pivot стоит на оригинальной земной плоскости (terrain: y=0, terrain_side: y∈[-0.5, 0])

**Проверка:** `tools/verify_recenter.gd` (headless) — замеряет AABB всех ассетов и репортит если центр XZ не в нуле (допуск 0.01).

```bash
"C:\Godot_v4.6.1-stable_win64_console.exe" --headless -s tools/verify_recenter.gd --path . 2>&1 | tail -10
# Ожидаемо: "Bad (center XZ not at 0): 0"
```

**Следствия для loader'а:** `map_loader.gd` не считает AABB runtime и не компенсирует пивот. `inst.position = cell_center`, `inst.rotation_degrees.y = rot` — и всё. Вращение вокруг Y естественно работает вокруг визуального центра, потому что он совпадает с локальным origin.

**Если добавлен новый .glb в папку:** убедиться что в его `.glb.import` прописан `import_script/path="res://tools/recenter_on_import.gd"`. Либо запустить reimport — Godot автоматически сгенерит .import файлы, а batch для новых можно сделать через sed.

## Формула размещения

```
world_pos = Vector3(x * tile_size, y_level * level_height, z * tile_size) + origin_offset
```

Где `y_level` — **высота в единицах level_height**. Для terrain — это его уровень. Для ramp/side — см. ниже.

## Y-rotation в Godot (для ramp/side ориентаций)

Godot Y-rotation: положительный угол = CCW при взгляде сверху (правая рука вокруг +Y).

- `rot=0` — без поворота
- `rot=90` — поворот CCW на 90° (было +X, стало +Z)
- `rot=180` — поворот на 180° (было +X, стало -X)
- `rot=-90` — поворот CW на 90° (было +X, стало -Z)

## Правила по типам ассетов

### `terrain` (плоская плитка)

- `y_level` = уровень на котором лежит плитка (0 = пол void, 1 = +0.5м, 2 = +1.0м…)
- `rot` = 0 (симметрична, ротация не нужна)
- ✅ **Проверено:** плитки стыкуются без зазоров.

### `terrain_ramp` (наклонная плитка)

- Геометрия уходит вверх на +level_height
- `y_level` = **низкий** уровень (к какому стыкуется низкий край)
- **rot=0** → подъём в +X ✅ проверено
- **rot=180** → подъём в −X (не проверено, должно работать)
- **rot=90** → подъём в −Z (не проверено)
- **rot=-90** → подъём в +Z (не проверено)
- Пример: ramp на (1, 0, 0) y_level=0 rot=0 — верх стыкуется с terrain на (2, 0, 0) y_level=1.

### `terrain_side` (наклонная плита — ramp)

**ВАЖНО:** `terrain_side` — это **плоский 4-угольник (quad)** с 4 вершинами, а не 3D клин со стенками. Две противоположные грани находятся на разных Y-уровнях (size.y=0.5), геометрия просто наклонена. Нижней закрывающей стенки нет — если это видно в кадре, под плато нужен terrain нижнего уровня.

**Raw vertices после recenter (in local space):**
- `(+0.5, 0, +0.5)` — высокая, +X +Z угол
- `(-0.5, 0, +0.5)` — высокая, -X +Z угол
- `(-0.5, -0.5, -0.5)` — низкая, -X -Z угол
- `(+0.5, -0.5, -0.5)` — низкая, +X -Z угол

То есть **высокий край = линия Z=+0.5** (от -X до +X, Y=0), **низкий край = линия Z=-0.5** (Y=-0.5).

**Направление "спуска"** = куда смотрит низкий край (из центра):

| rot  | Спуск в   | Высокий край на | Когда использовать (periphery плато)  |
|------|-----------|-----------------|---------------------------------------|
| 0    | **−Z**    | +Z              | Top border (z < plateau)              |
| 90   | **−X**    | +X              | Left border (x < plateau)             |
| 180  | **+Z**    | −Z              | Bottom border (z > plateau)           |
| -90  | **+X**    | −X              | Right border (x > plateau)            |

`y_level` для side = высокий уровень (тот к которому пристыковано плато).

✅ Проверено через vertex inspection в `test_plateau.json`.

### `terrain_sideCorner`, `terrain_sideCornerInner`, `terrain_sideEnd`, `terrain_sideCliff`

- Не протестированы. Требуют индивидуальной диагностики.
- `sideCorner` — внешний угол выпуклого плато
- `sideCornerInner` — внутренний угол вогнутого плато
- `sideEnd` — острый конец выступа (полуостров в 1 cell)
- `sideCliff` — крутой обрыв (возможно 1 уровень высоты вместо 0.5)

### `terrain_road*` (дороги)

- По форме идентичны `terrain` (плоские), отличаются только текстурой.

## Props (декоративные — камни, ракеты, здания)

Props ставятся в JSON массиве `"props"` с float-координатами:

```json
{"asset": "rock_largeA", "x": 3.5, "z": 2.0, "y_level": 1, "rot": 0, "scale": 1.0}
```

Имена: `rock.glb` — мелкая кочка, `rock_largeA.glb` — большой камень с плоской вершиной, `rocks_smallA/B.glb` — россыпь мелких камней, `rock_crystals.glb` и `rock_crystalsLargeA/B.glb` — кристаллы.

## JSON формат

```json
{
  "meta": {
    "tile_size": 1.0,
    "level_height": 0.5,
    "origin_offset": [-2.5, 0, -2.5]
  },
  "rect_fills": [
    {"asset": "terrain", "x_min": 1, "z_min": 1, "x_max": 4, "z_max": 4, "y_level": 1, "rot": 0}
  ],
  "cells": [
    {"asset": "terrain_side", "x": 5, "z": 1, "y_level": 1, "rot": 0}
  ],
  "props": [
    {"asset": "rock_largeA", "x": 3.0, "z": 2.5, "y_level": 1, "scale": 1.0}
  ]
}
```

- `rect_fills` — прямоугольник одним ассетом (удобно для плоских плато)
- `cells` — индивидуальные клетки с явным ассетом (для рамп, sides, углов)
- `props` — декор, float-координаты, без cell-логики
- Любой объект может содержать `"_": "комментарий"` — игнорируется loader'ом

### `origin_offset`

Сдвигает всю карту в world. Обычно ставлю так, чтобы центр карты был в `(0, 0, 0)`. Для карты `NxN`: `origin_offset = [-N/2, 0, -N/2]`.

## Loader

`scripts/map_loader.gd` — `@tool` скрипт. В Inspector кнопка **"Reload Map"** для перезагрузки без перезапуска игры.

- `@export var auto_load` — загружать при `_ready`
- `@export var generate_collision` — генерить collision shapes
- `@export_file var map_path` — путь к JSON

Loader **тупо размещает** ассеты по координатам — никакой auto-sides / auto-corner магии. Это **намеренно**: в прошлых версиях авто-логика создавала больше багов чем решала.

### Нюансы реализации

- **`add_child(inst, true)` без INTERNAL_MODE** — инстансы видны в scene panel редактора (можно выделить), не сохраняются в .tscn (регенерируются из JSON каждый раз)
- **`child.free()` в `_clear_children`** — не `queue_free()`! Deferred queue_free приводит к тому что на момент reload старые дети ещё не удалены, а новые уже добавлены

## Правила диагностики 3D-ассетов (важно!)

**Никогда не доверять только одному ракурсу** — особенно top-down. Плоские и 3D-формы выглядят одинаково сверху.

**Обязательные проверки при работе с новым ассетом или новой ориентацией:**

1. **Top-down** — проверить что XZ-позиции правильные (никаких смещений по plane)
2. **Низкий iso ракурс** (камера на y≈2-4, расстояние 6-8, смотрит на центр) — увидеть **наклоны** и высоты
3. **Вид прямо сбоку** (камера на одной высоте с объектом, смотрит горизонтально) — увидеть **профиль склона**
4. **Вид с низкой точки снизу-вверх** — проверить есть ли под-сторона у ассета (стенки или нет)

**При любых сомнениях про форму/ориентацию meshа — читать raw vertices напрямую:**

```gdscript
# через game_eval в MCP
var mi = get_node("path/to/MeshInstance3D")
var arrays = mi.mesh.surface_get_arrays(0)
var verts: PackedVector3Array = arrays[Mesh.ARRAY_VERTEX]
for v in verts:
    print("%.2f, %.2f, %.2f" % [v.x, v.y, v.z])
```

Это даст точное понимание геометрии без догадок по визуалу.

**Lesson learned (2026-04-23):** `terrain_side` выглядел на top-down как 3D клин, и я на доверии угадывал направление спуска по логике "клин как ramp". Оказалось, что это **плоская наклонная quad из 4 вершин**, и вся моя таблица rot→direction была неверной. Исправил только после vertex inspection. **Всегда проверять raw данные для новых ассетов.**

## Диагностические сцены

- `scenes/asset_diag.tscn` — все ассеты в сетку с подписями (визуальный каталог)
- `scenes/asset_test_ladder.tscn` — лесенка terrain → ramp → plateau → side → terrain (проверка стыков)
- `scenes/map_preview.tscn` — loads JSON из `map_path`, preview-режим без physics
- `resources/maps/test_4sides.json` — isolation test для 4 rotations
- `resources/maps/test_plateau.json` — 4×4 плато с 16 sides периметром

**Правило:** при любом новом типе ассета / rotation / стыка — сначала диагностический тест, потом включение в настоящую карту.

## Проверочный чеклист перед коммитом карты

1. `verify_recenter.gd` — все ассеты отцентрированы?
2. Все плоские плитки стыкуются без зазоров (вид сверху)?
3. Рампы/sides стоят на правильной Y (стыки без провалов/перехлёстов)?
4. Углы корректно закрыты (нет дыр в углах плато)?
5. Props не проваливаются под землю и не висят над ней?
6. Collision работает (игрок не проваливается / не застревает)?
