# Договорённости по рабочему процессу

Дата: 2026-03-26

Этот документ фиксирует проблемы, с которыми мы столкнулись при разработке с AI, и согласованные решения.

---

## Проблема 1: AI не может тестировать свои изменения

**Суть:** Godot — визуальный движок. AI не может запустить игру и проверить, что изменения работают. Каждый баг требует ручной проверки пользователем → цикл обратной связи очень медленный. Результат: баги ротации/физики тянулись несколько сессий.

**Решения:**

1. **Одно изменение = один тест.** AI не делает несколько изменений за раз. Сначала одно, пользователь тестирует, потом следующее.
2. **Чёткое описание ожидаемого результата.** При каждом изменении визуала/физики AI описывает ЧТО конкретно должно измениться, чтобы пользователь мог быстро проверить (например: "машинка теперь должна ехать носом вперёд, камера — сзади").
3. **Автоматическая проверка синтаксиса.** После каждого изменения скрипта — запускать `godot --headless --check-only` для ловли синтаксических ошибок.
4. **MCP Godot сервер.** Использовать когда доступен — даёт обратную связь о состоянии проекта в реальном времени.

---

## Проблема 2: Кодовые костыли вместо инструментов Godot

**Суть:** Вместо встроенных возможностей Godot (AnimationPlayer, Resource, @export, сигналы, EditorPlugin) AI писал ручные решения в коде. Это приводило к более сложному и хрупкому коду.

**Решения:**

1. **Всегда сверяться с документацией.** Перед реализацией любой фичи, связанной с движком (физика, анимации, сеть, UI), обязательно проверить Context7 docs.
2. **Предпочитать нативные решения:**
   - `Resource` вместо `Dictionary` для данных
   - `@export` вместо хардкода параметров
   - `AnimationPlayer` вместо ручного tween-кода
   - Сигналы вместо прямых вызовов между узлами
3. **Вопрос перед реализацией.** Прежде чем писать кастомный код — спросить: "есть ли встроенный способ в Godot?"

---

## Проблема 3: Беспорядок в проекте

**Суть:** Остались мёртвые файлы (kart.tscn, arena_builder.gd, arena_map.gd), устаревшая информация в CLAUDE.md, несоответствие описаний реальному состоянию кода.

**Решения:**

1. **Периодическая чистка.** Удалять неиспользуемые файлы, не накапливать мусор.
2. **Не создавать файлы без необходимости.** Предпочитать редактирование существующих файлов.
3. **Актуализировать CLAUDE.md.** При значительных изменениях архитектуры — обновлять описание ключевых файлов и стека.

---

## Общий принцип

**MVP first.** Не добавлять новые фичи, пока база не работает стабильно. Текущий приоритет:
1. Рабочее управление + физика карта
2. HTML5 export для браузера
3. Стабильный мультиплеер через ngrok
4. Базовое оружие
